   Data storage paradigm
   [https://en.wikipedia.org/wiki/File:BabbageKeyValueCard.tiff][lossless-
   page1-350px-BabbageKeyValueCard.tiff.png]
   [https://en.wikipedia.org/wiki/File:BabbageKeyValueCard.tiff]
   A tabular data card proposed for Babbage's
   [https://en.wikipedia.org/wiki/Analytical_Engine]Analytical Engine
   showing a key–value pair, in this instance a number and its base-ten
   logarithm

   A key–value database, or key–value store, is a data storage paradigm
   designed for storing, retrieving, and managing
   [https://en.wikipedia.org/wiki/Associative_array]associative arrays,
   and a [https://en.wikipedia.org/wiki/Data_structure]data structure more
   commonly known today as a dictionary or
   [https://en.wikipedia.org/wiki/Hash_table]hash table. Dictionaries
   contain a
   [https://en.wikipedia.org/wiki/Collection_(abstract_data_type)]collecti
   on of [https://en.wikipedia.org/wiki/Object_(computer_science)]objects,
   or [https://en.wikipedia.org/wiki/Record_(computer_science)]records,
   which in turn have many different
   [https://en.wikipedia.org/wiki/Field_(computer_science)]fields within
   them, each containing data. These records are stored and retrieved
   using a key that uniquely identifies the record, and is used to find
   the data within the [https://en.wikipedia.org/wiki/Database]database.
   [https://en.wikipedia.org/wiki/File:KeyValue.PNG][KeyValue.PNG]
   [https://en.wikipedia.org/wiki/File:KeyValue.PNG]
   A table showing different formatted data values associated with
   different keys

   Key–value databases work in a very different fashion from the better
   known [https://en.wikipedia.org/wiki/Relational_database]relational
   databases (RDB). RDBs predefine the data structure in the database as a
   series of tables containing fields with well defined
   [https://en.wikipedia.org/wiki/Data_type]data types. Exposing the data
   types to the database program allows it to apply a number of
   optimizations. In contrast, key–value systems treat the data as a
   single opaque collection, which may have different fields for every
   record. This offers considerable flexibility and more closely follows
   modern concepts like
   [https://en.wikipedia.org/wiki/Object-oriented_programming]object-orien
   ted programming. Because optional values are not represented by
   placeholders or input parameters, as in most RDBs, key–value databases
   often use far less
   [https://en.wikipedia.org/wiki/Computer_memory]memory to store the same
   data, which can lead to large performance gains in certain
   workloads.^[[https://en.wikipedia.org/wiki/Wikipedia:Citation_needed]ci
   tation needed]

   Performance, a lack of standardization and other issues limited
   key–value systems to niche uses for many years, but the rapid move to
   [https://en.wikipedia.org/wiki/Cloud_computing]cloud computing after
   2010 has led to a renaissance as part of the broader
   [https://en.wikipedia.org/wiki/NoSQL]NoSQL movement. Some
   [https://en.wikipedia.org/wiki/Graph_database]graph databases, such as
   [https://en.wikipedia.org/wiki/ArangoDB]ArangoDB,^[https://en.wikipedia
   .org/wiki/Key-value_database#cite_note-1][1] are also key–value
   databases internally, adding the concept of the relationships
   ([https://en.wikipedia.org/wiki/Pointer_(computer_programming)]pointers
   ) between records as a first class data type.

Types and
examples[[https://en.wikipedia.org/w/index.php?title=Key%E2%80%93value_databa
se&action=edit&section=1]edit]

   Key–value databases can use
   [https://en.wikipedia.org/wiki/Consistency_model]consistency models
   ranging from
   [https://en.wikipedia.org/wiki/Eventual_consistency]eventual
   consistency to
   [https://en.wikipedia.org/wiki/Serializability]serializability. Some
   support ordering of keys.

   Some maintain data [https://en.wikipedia.org/wiki/In-memory_database]in
   memory (RAM), while others employ
   [https://en.wikipedia.org/wiki/Solid-state_drive]solid-state drives or
   [https://en.wikipedia.org/wiki/Hard_disk_drive]rotating disks.

   Every entity (record) is a set of key–value pairs. A key has multiple
   components, specified as an ordered list. The major key identifies the
   record and consists of the leading components of the key. The
   subsequent components are called minor keys. This organization is
   similar to a directory path specification in a file system (e.g.,
   /Major/minor1/minor2/). The “value” part of the key–value pair is
   simply an uninterpreted string of bytes of arbitrary
   length.^[https://en.wikipedia.org/wiki/Key-value_database#cite_note-2][
   2]

   The Unix system provides
   [https://en.wikipedia.org/wiki/DBM_(computing)]dbm (database manager),
   which is a 1979 library originally written by
   [https://en.wikipedia.org/wiki/Ken_Thompson]Ken Thompson. It is also
   ported to [https://en.wikipedia.org/wiki/Microsoft_Windows]Microsoft
   Windows, provided through programming languages such as
   [https://en.wikipedia.org/wiki/Perl#Windows]Perl for Win32. The dbm
   manages associative arrays of arbitrary data by use of a single key (a
   primary key). Modern implementations include
   [https://en.wikipedia.org/wiki/DBM_(computing)#Implementations]sdbm,
   GNU dbm, and [https://en.wikipedia.org/wiki/Berkeley_DB]Berkeley DB.
   Although dbm precedes the concept of a NoSQL and is rarely mentioned in
   modern discourse, it is used by many pieces of software.

   A more recent example of a key-value database is
   [https://en.wikipedia.org/wiki/RocksDB]RocksDB which is used as storage
   engine for other database management systems such as
   [https://en.wikipedia.org/wiki/ArangoDB]ArangoDB.
